# Challenge Lifecycle

Challenges in FCTF move through visibility, deployment, and solve states.

This page summarizes the lifecycle as seen by contestants and enforced by backend services.

## Lifecycle Stages

| Stage | Contestant-visible behavior | Typical trigger |
| --- | --- | --- |
| Hidden | Challenge does not appear in list | Admin state is hidden or requirement policy hides it |
| Locked | Challenge card exists but cannot be solved yet | Prerequisite not completed |
| Available | Challenge detail is accessible | Visibility and requirements satisfied |
| Pending Deploy | Start requested, instance not ready yet | Deploy-required challenge startup |
| Running | Token and access endpoint available | Runtime health checks passed |
| Solved | Challenge marked solved for team | Correct flag submission |
| Stopped | Instance terminated, token no longer usable | Manual stop, auto-stop timeout, or lifecycle cleanup |

## Visibility Rules

- **Visible + unlocked**: challenge appears with full metadata and actions.
- **Locked by requirements**: contestant sees lock indicators and prerequisite hints.
- **Anonymized by policy**: challenge can be listed with limited metadata (policy-dependent).

## Prerequisite Semantics

A prerequisite chain controls progression and prevents sequence bypass.

1. Team solves required upstream challenge(s).
2. Backend recalculates eligibility.
3. Newly unlocked challenges become available or de-anonymized.

## Deploy Runtime States

For challenges with `require_deploy=true`, the lifecycle includes runtime status transitions:

1. Contestant clicks `[+] Start Challenge`.
2. System enters `Pending/Checking` while deployment is prepared.
3. On success, `[YOUR ACCESS TOKEN]` is shown with protocol-specific endpoint.
4. Instance remains running until one of these events:
	- contestant presses `[-] Stop Challenge`
	- challenge timer expires (auto-stop)
	- backend cleanup/reconciliation terminates stale runtime

The same instance state is reflected in the contestant `Instances` page as `Pending`, `Running`, or deletion-related terminal states.

## Submission Interaction with Lifecycle

- Correct submission transitions challenge state to `Solved` for that team.
- Incorrect submissions consume attempts and may trigger cooldown.
- When attempt quota is exhausted, submission becomes blocked for that challenge/team pair.
- For deploy-required challenges, runtime can be stopped automatically after solve depending on challenge flow.

## Why This Matters

- Preserves intended progression and competition pacing.
- Keeps scoring integrity under concurrent solves.
- Reduces resource waste through explicit runtime stop conditions.
- Gives contestants predictable status transitions during live events.
