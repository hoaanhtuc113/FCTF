# Overview

## Document Scope

This document describes the implemented architecture of the FCTF platform as a self-hosted, on-premise CTF system running on k3s.

It focuses on:

- Real service boundaries and runtime behavior in this repository.
- How challenge environments are built, deployed, accessed, and reconciled.
- Security, storage, scaling, and operations decisions with trade-offs.

It does not attempt to be a generic Kubernetes guide.

<Tabs groupId="architecture-overview">
  <TabItem value="foundation" label="Foundation" default>
    <ul>
      <li>On-premise, self-hosted k3s deployment.</li>
      <li>Services are split into access, control, execution, and reconciliation planes.</li>
      <li>Challenge runtime is governed by explicit network, storage, and lifecycle controls.</li>
    </ul>
  </TabItem>
  <TabItem value="runtime" label="Runtime Lifecycle">
    <ul>
      <li>Challenge start is asynchronous: DeploymentCenter -&gt; RabbitMQ -&gt; DeploymentConsumer -&gt; Argo.</li>
      <li>Challenge instances are isolated per namespace with restrictive NetworkPolicies.</li>
      <li>Runtime sandboxing is available via gVisor (`runtimeClassName: gvisor`) per deployment policy.</li>
      <li>ChallengeGateway provides controlled HTTP/TCP access using signed tokens.</li>
    </ul>
  </TabItem>
  <TabItem value="ops" label="Ops and Reliability">
    <ul>
      <li>DeploymentListener continuously reconciles cluster state with platform state.</li>
      <li>Redis Lua scripts enforce quota, TTL, and race-safe state transitions.</li>
      <li>Prometheus, Grafana, and Loki provide health, metrics, and log traceability.</li>
    </ul>
  </TabItem>
</Tabs>
